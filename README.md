# Киберимунная RFID Система Контроля Доступа (Шлагбаум)

## Описание проекта

 Реализует систему контроля доступа на основе RFID-карт с применением принципов **киберимунности** , архитектуры **Zero Trust** и концепции **Доверенной Вычислительной Базы** (TCB).

---

## Принципы киберимунности

| Принцип | Реализация |
|---|---|
| Доверенная Вычислительная База (TCB) | Модуль `core/` — единственный доверенный компонент |
| Разделение доверенных/недоверенных зон | `core/` (trusted) vs `devices/`, `api/` (untrusted) |
| Единая точка принятия решений | `PolicyEngine` — один компонент оценивает все запросы |
| Единый процессор запросов | `AccessController` — все запросы проходят через него |
| Принцип наименьших привилегий | Трёхуровневая модель доступа (user/operator/admin) |
| Внедрение зависимостей через интерфейсы | Все компоненты взаимодействуют через ABC-интерфейсы |
| Минимальное связывание | Каждый слой зависит только от интерфейсов нижнего слоя |
| Zero Trust | Каждый запрос проверяется заново, сессии не сохраняют разрешения |

## Архитектура

```
api/          ← Публичный интерфейс (недоверенная зона)
  ├── access_api.py   — обработка считываний RFID
  ├── admin_api.py    — управление пользователями и картами
  ├── monitor_api.py  — мониторинг состояния системы
  └── operator_api.py — ручное управление шлагбаумом

services/     ← Бизнес-логика (прослойка между API и ядром)
  ├── access_service/
  ├── administration_service/
  ├── operator_service/
  └── user_service/

core/         ← ДОВЕРЕННАЯ ВЫЧИСЛИТЕЛЬНАЯ БАЗА
  ├── access_controller/  — единый процессор запросов
  ├── policy_engine/      — единая точка принятия решений
  ├── audit/              — неизменяемый журнал аудита
  ├── database/           — репозиторий данных
  └── security_monitor/   — детектор аномалий

devices/      ← Физические устройства (недоверенная зона)
  ├── barrier_driver/  — привод шлагбаума
  ├── rfid_reader/     — считыватель RFID
  ├── traffic_light/   — светофор
  ├── operator_panel/  — панель оператора
  └── modem/           — модем для оповещений

security/     ← Криптография, аутентификация, авторизация, сессии
entities/     ← Доменные объекты (датаклассы и перечисления)
interfaces/   ← Абстрактные интерфейсы (контракты взаимодействия)
communication/← Транспортный слой (Request/Response/Command/Event)
config/       ← Конфигурация системы
tests/        ← Тестовый набор
```

### Поток обработки запроса

```
RFID Reader → AccessAPI → AccessService → AccessController
                                               ↓
                                         PolicyEngine → AccessRepository
                                               ↓
                                         BarrierDriver (открыть/закрыть)
                                               ↓
                                    SecurityMonitor + AuditLogger
```

## Установка и запуск

### 1. Клонирование / распаковка архива

```bash
unzip barrier-system.zip
cd barrier-system
```

### 2. Создание виртуального окружения (рекомендуется)

```bash
python -m venv .venv
source .venv/bin/activate      # Linux / macOS
.venv\Scripts\activate.bat     # Windows
```

### 3. Установка зависимостей

```bash
pip install -r requirements.txt
# или через make:
make install
```

### 4. Настройка переменных окружения (опционально)

```bash
cp .env.example .env
# отредактируйте .env при необходимости
```

### 5. Запуск системы

```bash
python start.py
# или через make:
make run
```

При запуске с `SEED_DATA=true` (по умолчанию) автоматически создаются тестовые данные и выполняется демонстрация четырёх сценариев:

1. Действительная карта → доступ **разрешён**, шлагбаум открывается
2. Неизвестная карта → доступ **запрещён**
3. Заблокированная карта → доступ **запрещён**
4. Повторный проход (выезд) → доступ **разрешён**

### 6. Запуск тестов

```bash
pytest tests/ -v
# или через make:
make test

# с отчётом покрытия:
make test-cov
```

## Переменные окружения

| Переменная | По умолчанию | Описание |
|---|---|---|
| `BARRIER_ID` | `barrier-01` | Идентификатор шлагбаума |
| `READER_ID` | `reader-01` | Идентификатор считывателя |
| `MIN_ACCESS_LEVEL` | `1` | Минимальный уровень доступа |
| `OPERATOR_ACCESS_LEVEL` | `2` | Уровень доступа оператора |
| `ADMIN_ACCESS_LEVEL` | `3` | Уровень доступа администратора |
| `LOOP_INTERVAL` | `0.5` | Интервал основного цикла (сек) |
| `DEBUG` | `false` | Режим отладки |
| `SEED_DATA` | `true` | Загрузка тестовых данных при старте |
| `CRYPTO_ALGORITHM` | `sha256` | Алгоритм хеширования |
| `SESSION_TIMEOUT` | `3600` | Тайм-аут сессии (сек) |
| `MAX_FAILED_ATTEMPTS` | `5` | Макс. число неудачных попыток |

## Документация API

### AccessAPI — Обработка RFID

| Метод | Аргументы | Описание |
|---|---|---|
| `process_card_read(uid, reader_id, direction)` | `uid: str`, `reader_id: str`, `direction: str` | Обработать считывание карты |
| `get_barrier_state()` | — | Получить текущее состояние шлагбаума |
| `open_barrier(caller_id)` | `caller_id: str` | Открыть шлагбаум |
| `close_barrier(caller_id)` | `caller_id: str` | Закрыть шлагбаум |

### AdminAPI — Управление (уровень доступа ≥ 3)

| Метод | Описание |
|---|---|
| `add_user(admin, user)` | Добавить пользователя |
| `remove_user(admin, identifier)` | Удалить пользователя |
| `block_user(admin, identifier)` | Заблокировать пользователя |
| `unblock_user(admin, identifier)` | Разблокировать пользователя |
| `register_card(admin, card)` | Зарегистрировать RFID-карту |
| `revoke_card(admin, uid)` | Отозвать RFID-карту |
| `list_users(admin)` | Список пользователей |
| `list_cards(admin)` | Список карт |
| `get_audit_log(admin)` | Журнал аудита |
| `emergency_open(admin, access_service)` | Аварийное открытие |
| `emergency_close(admin, access_service)` | Аварийное закрытие |

### MonitorAPI — Мониторинг (уровень доступа ≥ 2)

| Метод | Описание |
|---|---|
| `get_system_status(user)` | Общий статус системы |
| `get_barrier_state(user)` | Состояние шлагбаума |
| `get_active_sessions(user)` | Активные сессии |
| `get_security_events(user)` | Поток событий безопасности |
| `get_anomaly_status(user)` | Статус детектора аномалий |
| `health_check()` | Проверка работоспособности (без авторизации) |

### OperatorAPI — Панель оператора (уровень доступа ≥ 2)

| Метод | Описание |
|---|---|
| `authenticate_operator(operator)` | Аутентифицировать оператора |
| `open_barrier(operator)` | Открыть шлагбаум |
| `close_barrier(operator)` | Закрыть шлагбаум |
| `get_barrier_state(operator)` | Состояние шлагбаума |
| `get_access_history(operator)` | История доступа |

## Структура кода

Проект строго следует принципу **единственной ответственности**: каждый класс выполняет ровно одну задачу. Бизнес-логика не дублируется — API делегирует в Services, Services вызывают Core, Core управляет устройствами через интерфейсы.
